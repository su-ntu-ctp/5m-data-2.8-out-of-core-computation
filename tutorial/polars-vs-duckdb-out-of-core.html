Polars vs DuckDB

Both can crunch datasets larger than RAM — but they get there differently. Polars streams data through a pipeline in fixed-size morsels. DuckDB uses a buffer pool that spills hot pages to a temp file when memory fills up. Move the sliders below to see when each strategy wins.

2 · What just happened?

Polars — streaming pipeline

With collect(engine="streaming") (or the older streaming=True), Polars splits the query into operators that pass morsels (small Arrow batches, typically a few MB) through a pipeline. Memory stays near-flat: only the morsels currently in flight plus any operator state (e.g. a hash table for group-by) live in RAM.

  • Best at: filter, projection, group-by with bounded cardinality, streaming joins on a small side.
  • Struggles when: the operator must see all data at once — global sort, large × large hash join, window over the full table. These are blocking operators; the streaming engine may fall back to in-memory or fail.
  • Mental model: a conveyor belt. If every station can act on one box at a time, the line never backs up.

DuckDB — buffer pool spills to disk

DuckDB allocates a buffer pool (default ≈ 80% of RAM, configurable via SET memory_limit). Hash tables, sort runs, and join build sides live as pages in that pool. When the pool is full, the buffer manager spills cold pages to a temp file (SET temp_directory) and reads them back on demand.

  • Best at: any SQL — including the hard cases. Out-of-core hash join, external merge sort, and window functions are all built in.
  • Cost: when working set > RAM, you pay disk I/O. Performance degrades gracefully instead of OOM-ing.
  • Mental model: a library with a small reading room. Books you're using sit on the table; the rest live on shelves you walk to.

3 · Same query, two engines

A 50 GB Parquet file. Group by customer_id, sum revenue, keep top 100.

Polars — lazy + streaming

import polars as pl

q = (
    pl.scan_parquet("sales/*.parquet")        # lazy, no read yet
      .filter(pl.col("year") == 2025)
      .group_by("customer_id")
      .agg(pl.col("revenue").sum())
      .sort("revenue", descending=True)
      .head(100)
)

# Streaming engine: morsel-by-morsel execution
top = q.collect(engine="streaming")

DuckDB — SQL on a file

import duckdb

con = duckdb.connect()
con.execute("SET memory_limit = '8GB'")
con.execute("SET temp_directory = '/tmp/duck_spill'")

top = con.execute("""
    SELECT customer_id, SUM(revenue) AS revenue
    FROM 'sales/*.parquet'
    WHERE year = 2025
    GROUP BY customer_id
    ORDER BY revenue DESC
    LIMIT 100
""").df()

4 · Trade-offs at a glance

DimensionPolarsDuckDB
OOC mechanism Streaming pipeline of morsels; near-flat memory Buffer pool + automatic spill to temp files
API DataFrame / lazy expression DSL (Python, Rust, JS) SQL (plus Python/R/Java client APIs)
Blocking ops (sort, large join, window) Limited — streaming engine handles some; may fall back to in-memory or error Fully supported out-of-core (external sort, partitioned hash join)
Memory limit Implicit; set process limits / control morsel size Explicit: SET memory_limit = '8GB'
Spill location No general spill — relies on pipelining Configurable temp dir; spills hash tables & sort runs
Parquet / CSV scan Predicate & projection pushdown, partition pruning Same — both query files directly, no full load
Sweet spot ETL, feature engineering, DataFrame workflows in Python Ad-hoc SQL analytics on data > RAM; complex joins; embedded in apps
When it hurts Heavy joins or sorts on data ≫ RAM Disk-bound when working set far exceeds RAM (still works, but I/O slow)

5 · Rule of thumb

Use Polars when…

  • You're in a Python ML / ETL pipeline.
  • Your query is mostly filter / project / group-by.
  • You want a DataFrame, not a connection.

Use DuckDB when…

  • You think in SQL.
  • You need joins / sorts / windows on huge data.
  • You want graceful degradation, not OOM.

Use both

  • They share Apache Arrow under the hood.
  • Hand a pl.DataFrame to DuckDB with zero copy: duckdb.sql("FROM df").
  • Pick the engine per step of the pipeline.
